# Transport Simulation Game - Design Document

## ビジョン

simutransの「日本レベルの大路線建設」をもっと手軽に楽しめるゲーム。

幹線から枝線、貨物線、貨物ターミナル、バス、高速道路、会社間競争、運転種別分離...
これらを**少ないクリック数で**構築し、制約によるリアリティのある都市の発展を楽しむ。

## コンセプト

### simutransとの違い

| simutrans | 本作 |
|-----------|------|
| 1マスずつ敷設 | 駅を置いて繋ぐだけ |
| 単線/複線を手動管理 | 常に複線。閉塞は自動 |
| 信号を手動配置 | 自動配置 |
| 配線を手動設計 | 駅のホーム数を指定するだけ |
| 操作に時間がかかる | 路線設計と経済判断に集中 |

### プレイヤーが判断すること

- **どの都市を結ぶか**（路線網の設計）
- **何を運ぶか**（旅客、貨物、加工チェーン）
- **列車の頻度**（本数と編成長）
- **駅の容量**（ホーム数の増設判断）
- **幹線と枝線の階層構造**（フィーダー輸送）
- **ボトルネックの解消**（需要増加への対応）

### プレイヤーが判断しないこと（自動化）

- 信号配置
- 線路の単線/複線（常に複線）
- 配線の詳細
- ポイント切替

## 核心ルール: 需要は無限に増える

輸送量に応じて需要が増大する。需要が増えると:
- 都市が成長（人口増加、建物増加、市域拡大）
- 産業の生産量が増加
- 旅客が増加
- ボトルネックが発生（駅の容量不足、路線の本数不足）

プレイヤーは常にボトルネックを発見し、解消する。
解消するとさらに需要が増え、新たなボトルネックが生まれる。
この無限ループがゲームの核。

## アーキテクチャ

### タイルベース + グラフ操作

- **世界・表示**: タイルベース（地形、建物、線路）
- **操作**: グラフレベル（ノード配置→エッジ接続→タイル自動展開）
- **シミュレーション**: 閉塞システム + 経済システム

### 物理レールと路線の分離

- **物理レール（グラフ）**: ノード（駅）とエッジ（線路）で構成。共有インフラ
- **路線（Route）**: 物理レール上の運行計画。停車駅リスト + 運行モード

## グラフモデル

### ノード = 駅

駅のみ。信号場・信号ノードは不要（複線のみなので）。

- **接続数**: 最大2方向。1方向=終端駅（折り返し）、2方向=通過駅
- **容量**: ホーム数。同時に停車できる列車数
- 容量増設でボトルネック解消（コストがかかる）

### 駅複合体（転線）

隣接するタイル（チェビシェフ距離1）に配置された駅は**駅複合体**を形成する。
連鎖的な隣接もまとめて1つの複合体になる（A隣接B、B隣接C → A,B,Cは同一複合体）。

- **列車の転線**: 複合体内の駅間を即座に移動可能（暗黙のリンク、エッジ不要）
- **旅客の乗り換え**: 複合体内の駅間で旅客が移動可能（将来実装）
- **BFS経路探索**: エッジと転線の両方を探索し、最短経路を決定
- **分岐の表現**: 駅を隣接配置するだけで分岐を構築。各駅は2方向制限のまま

```
例: 分岐駅
A ── B1   B2 ── C     メインライン（B1-B2は隣接→転線可能）
      B3               B3もB1,B2に隣接
      │
      D                支線
```

### エッジ = 複線路

- 常に複線。同方向に複数列車が走行可能
- 閉塞は自動（エッジ内信号を自動配置）
- A*経路探索で地形コストを考慮した最適経路を自動計算

### 駅の容量

- 初期容量: 2（2ホーム）
- 増設可能（コスト増）
- 容量 = 同時に停車できる列車数
- 容量超過 → 列車がエッジ上で待機 → ダイヤ遅延 → ボトルネック

## 閉塞システム（キューベース・複線専用）

### 概念

- **セクション = 閉塞区間**: エッジ内で自動的に等間隔（10タイル）で区切られる。方向別に独立
- **ノード = キュー**: スロット（容量分）+ 待機列。列車は拒否されず常にキューに入る
- 信号はパス長に応じて自動配置。プレイヤーは触らない

### ルール

1. 各セクションに同方向1台のみ。対向方向は独立（複線）
2. ノードはキュー: 容量分の先頭がスロット（出発可能・貨物積み下ろし可能）、残りは待機列
3. tryDepart: スロット内 AND セクション空き → 出発
4. arrive: 常に成功（キューに入る）。非停車ノードでは即座に次エッジへ乗り換え試行
5. 信号通過: 次セクションが空いていれば移動、塞がっていれば信号手前で停止

### 更新フェーズ

1. **フェーズ1（出発）**: AtNode列車がtryDepartを試みる
2. **フェーズ2（移動）**: OnEdge列車が移動、信号通過、到着

## 路線（Route）

### 構造

- **name**: 路線名
- **stops**: 停車駅ノードIDの順序リスト
- **mode**: Shuttle（折り返し）/ Loop（巡回）

### 列車の経路決定

BFSでRouteの次の停車駅への最短経路を探索。

## 経済システム

### 需要と供給の連鎖

```
農場 → [米] → 商業施設 → [日用品] → 住宅
鉱山 → [鉄] → 工場 → [製品] → 住宅
住宅 → [旅客] → 住宅（他都市）
```

### 需要増大ループ

1. 輸送が成立する → 都市が成長
2. 都市が成長 → 人口増加、産業拡大
3. 需要が増大 → ボトルネック発生
4. プレイヤーがボトルネック解消（増便、駅増設、新路線）
5. → 1に戻る

### 都市

- **市域**: 中心座標 + 半径。成長で拡大
- **建物**: タイル単位。住居（市域内）、産業（市域外）
- **人口**: 市域内の住居の合計
- 住居は消費財の供給に応じて増加

## 平面交差

- エッジのパスが他のエッジのパスと同じタイルを通過することを許可する
- ゲーム上は**立体交差**（高架/地下）として扱う
- ノードは独立なので閉塞は干渉しない（列車はすり抜ける）
- 交差タイル数に応じて**建設コストが追加**される
- 駅はエッジパス上に建設できない（交差点に駅は置けない）

## 地形

- ノイズベースの地形生成
- 平地、山岳、水域
- 山岳: 建設コスト高、速度制限
- 水域: 通行不可（橋は将来実装）

## アーキテクチャ方針: ECS導入

### 背景

今後の機能追加で以下のエンティティ/コンポーネントが増える:

- **交通モード**: Train, Bus, Truck → 共通の Vehicle に Position, Route, Cargo コンポーネント
- **旅客**: Passenger エンティティに Origin, Destination, Preference（行動パターン・属性）
- **建物**: Building エンティティに Position, Production, Consumption, Population を動的に組み合わせ
- **乗り換え**: ノードでの Passenger → Vehicle マッチング

### 方針

- **miniplex**（TypeScript親和性の高い軽量ECSライブラリ）を採用
- 現在のクラスベースのデータ構造をECSのエンティティ + コンポーネントに移行
- システム（update関数）がコンポーネントをクエリして処理
- テスト時はWorldを自由に組み立て可能（シングルトン不使用）

### 移行対象

| 現在 | ECS後 |
|------|-------|
| Train interface | Entity + Position + Route + Cargo + TrainState コンポーネント |
| GraphNode | Entity + Position + NodeCapacity + Name コンポーネント |
| GraphEdge | Entity + EdgePath + Signals コンポーネント |
| Building | Entity + Position + Production + Consumption + Population コンポーネント |
| City | Entity + CityArea + Name コンポーネント |

### 移行タイミング

Step 10（需要増大ループ）の前に実施。旅客・建物の属性が増える前にECS基盤を整える。
