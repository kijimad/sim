# Transport Simulation Game - Design Document

## コンセプト

simutransの戦略的深さを維持しつつ、操作の煩雑さを解消する。

核心: 「現実を模倣するといい結果になる」ゲーム性を守る。

### 残すべき戦略要素

- **フィーダー輸送**: 支線で集めて幹線で運ぶ。路線設計に階層構造が生まれる
- **折り返し運転**: 終端駅での折り返し、途中駅での区間運転
- **需要に応じた複線化**: 単線で始めて、需要増加に合わせて段階的に複線化する判断
- **複数ホームの交互運用**: 高頻度発着を実現するための駅設計の工夫

これらが「知っていれば有利、知らなくても遊べる」深さを生む。

### simutransの問題点

- 1マスずつの敷設が煩雑(長距離路線、複線化)
- 物理レールのグルーピングができず管理が困難
- 自由度の高さが操作コストに直結

## アーキテクチャ

### タイルベース + グラフ操作

- **世界・表示・シミュレーション**: タイルベース
- **操作**: 常にグラフレベル(ノード配置→エッジ接続→タイルに自動展開)
- **1タイル単位の手動操作は提供しない**

グラフが正のデータ。タイル上の線路はグラフから導出される。細かい調整はノードの追加・移動で行う。

### 物理レールと路線の分離

独立した2つの概念:

- **物理レール(グラフ)**: 地形上に敷設されたインフラ。ノードとエッジで構成。共有資源
- **路線(Route)**: 物理レール上に定義された運行計画。停車駅リスト + 運行モード

1本のレール上に複数の路線が走る。

## グラフモデル

### ノード種別

駅と信号場はシミュレーション上は同一(容量を持つノード)。違いは客扱いの有無のみ。

| 種別 | デフォルト容量 | 客扱い | 用途 |
|------|--------------|--------|------|
| 駅 | 2 | あり | 旅客・貨物の乗降 |
| 信号場 | 2 | なし | 単線でのすれ違い |
| 信号 | 1 | なし | 閉塞区間の区切り |

### エッジ

- ノード間を接続する線路
- A*経路探索で地形コストを考慮した最適経路を自動計算
- エッジ上にタイルパスを持つ(描画・移動用)
- エッジ分割: 既存エッジの途中にノードを挿入して2つに分割可能

### 大きな駅の表現

1ノード = 1ホーム(capacity 1-2)。大きな駅は複数の隣接ノードで構成する。

- 隣接する駅ノード間は乗り換え可能(旅客がホーム間を移動)
- 各ホームにどの路線を引くかはプレイヤーが配線で設計する
- ホーム数を増やすには物理的にノードを追加し、エッジを引く必要がある → 空間を消費

```
例: 4面の大駅

       ──S1──    (東西本線 上り)
  A方面──S2──B方面 (東西本線 下り)
       ──S3──    (南北線)
       ──S4──    (鉱山支線)

S1-S4は隣接 → 乗り換え可能
```

### 駅の内部構造(複合ノード)

駅は1ノードだが、内部にホーム数・接続方向の情報を持つ。ホーム数に応じてタイル上の占有面積が増える。

```ts
interface StationLayout {
  orientation: "horizontal" | "vertical";
  platforms: number;       // ホーム数 → 占有幅が決まる
  assignments: Map<platformIndex, edgeId[]>; // 各ホームの接続先エッジ
}
```

- エッジを駅に繋ぐと、ホームへの割り当てをUIで設定
- ホーム割り当て変更 → エッジ端点タイルが移動 → パス自動再計算

### 配線の自動最適化

エッジのホーム割り当て時に交差を最小化する。

- 左右の接続先エッジをY座標でソートし、上から順にホーム割り当て(二部グラフ交差最小化)
- 不可避な立体交差にはコスト加算(建設費・速度制限)
- プレイヤーによる手動上書きも可能

### 将来拡張(未実装)

- エッジ属性: 線数(単線/複線)、電化有無、速度プロファイル
- ホーム長、ホームごとの入場条件
- 車両の編成長
- 空間的制約(タイルレベルの衝突判定)
- 駅の内部構造・配線最適化

## シミュレーションモデル

### 予約ベース(現在の実装)

列車の状態は2つだけ:

- **AtNode(nodeId)**: ノードに停車中
- **OnEdge(edgeId)**: エッジ上を移動中

### ルール

1. **AtNode → OnEdge**: 次のエッジが未予約なら予約して出発
2. **OnEdge → AtNode**: 端に到着時、ノード容量があり次エッジが空いていれば入る
3. **ノード容量 > 1の場合**: 次エッジが空いていなくてもノードに入れる(信号場での退避)
4. **ノード満杯 or 次エッジ占有(容量1ノード)**: エッジ上で停止して待つ

### エッジ予約

- `Map<edgeId, trainId>`: 1エッジに同時に1列車のみ
- 列車は出発時にエッジを予約、到着時に解放

### ノード占有

- `Map<nodeId, Set<trainId>>`: 容量まで複数列車を収容
- 容量超過の場合、到着列車はエッジ上で待機

## 路線(Route)

### 構造

- **stops**: 停車駅ノードIDの順序リスト
- **mode**: Shuttle(折り返し) / Loop(巡回)

### 運行モード

- **Shuttle**: A → B → C → B → A → ... (末端で折り返し)
- **Loop**: A → B → C → A → B → C → ... (環状運転)

### 列車の経路決定

列車はRouteのstopsに従い、次の目的駅に向かう最寄りのエッジを選択。
目的駅に到着するとRouteの次のstopに進む。

## 敷設操作

### 基本フロー

1. マップ上にノード(駅)を配置
2. ノード間を接続 → 地形考慮の最適経路が自動計算
3. 確定 → タイルに展開
4. 経路を変えたければ中間にノードを追加

### 信号場の挿入

信号場モードで既存エッジ付近をクリック → エッジを自動分割してノード挿入。

```
Before: A ────────────── B (1エッジ = 1閉塞)
After:  A ───── S ───── B (2エッジ = 2閉塞)
```

## 地形

- 建築・配置・運転を制約する
- ノイズベースの地形生成(2オクターブ、シード指定可能)
- 地形タイプ: 平地、山岳、水域
- 経路探索で地形コストを考慮(平地: 1、山岳: 5、水域: 通行不可)
